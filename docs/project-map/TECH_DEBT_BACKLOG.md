# NiTorrent — TECH_DEBT_BACKLOG

## Как пользоваться этим backlog
Это не просто список "что бы улучшить", а рабочий реестр технического долга для рефакторинга без потери контроля.

Приоритеты:
- **Critical** — высокий риск багов, потери данных, нестабильных релизов
- **High** — сильно мешает сопровождению, онбордингу и безопасным изменениям
- **Medium** — улучшает читаемость, тестируемость и предсказуемость
- **Low** — полезная полировка, но не должна тормозить основные этапы

Статусы можно вести так:
- `todo`
- `in-progress`
- `done`
- `wont-fix-now`

Рекомендуемый формат работы:
1. брать в спринт только 1–2 крупных пункта уровня Critical/High
2. каждую задачу закрывать отдельным PR или серией маленьких PR
3. не смешивать техдолг с большой feature-разработкой в одном изменении
4. обновлять этот файл после каждого заметного архитектурного изменения

## Critical

### TD-001. Перегруженный `MonoTorrentService`
**Статус:** todo  
**Файлы:** `src/NiTorrent.Infrastructure/Torrents/MonoTorrentService.cs`

**Проблема**
В одном классе сосредоточены слишком разные обязанности:
- lifecycle торрент-движка
- восстановление состояния
- add/start/pause/stop/remove
- snapshot mapping
- работа с синхронизацией
- часть orchestration-логики

**Риски**
- высокий шанс регрессий при любой правке
- сложная локализация причин багов
- слабая тестируемость
- тяжёлый онбординг в центральный кусок системы

**Что делать**
Разделить на отдельные роли:
- `TorrentEngineLifecycle`
- `TorrentCommandExecutor`
- `TorrentSnapshotFactory`
- `TorrentRuntimeRegistry`
- `TorrentStartupRecovery`
- `TorrentStateRepository`

**Ожидаемый эффект**
Снижение связности, меньше риск багов, проще тестирование и ревью.

---

### TD-002. Слишком много ответственности в `App.xaml.cs`
**Статус:** todo  
**Файлы:** `src/NiTorrent.App/App.xaml.cs`

**Проблема**
В точке входа смешаны:
- startup
- DI/bootstrap
- activation handling
- window lifecycle
- tray lifecycle
- shutdown orchestration
- запуск торрент-движка

**Риски**
- нестабильный startup/shutdown
- тяжело сопровождать релизные изменения
- сложно быстро понять flow приложения

**Что делать**
Вынести в отдельные сервисы:
- `AppBootstrapper`
- `AppActivationService`
- `WindowCoordinator`
- `ShutdownCoordinator`

**Ожидаемый эффект**
Более предсказуемый старт приложения и более простой онбординг.

---

### TD-003. Нет явного набора автоматических safety checks на критические сценарии
**Статус:** todo  
**Область:** startup, restore state, add/start/pause/remove, shutdown

**Проблема**
Критичные потоки завязаны на ручную проверку и знание текущего поведения.

**Риски**
- регрессии всплывают поздно
- рефакторинг становится опасным
- релизы сложно стабилизировать

**Что делать**
Минимум:
- выделить smoke/integration test plan
- покрыть хотя бы основные сценарии contract/integration проверками
- фиксировать текущее поведение перед изменениями

**Ожидаемый эффект**
Рефакторинг становится управляемым, а релизы — предсказуемее.

---

### TD-004. Логика восстановления состояния и сохранения слишком тесно связана с runtime
**Статус:** todo  
**Файлы:** `MonoTorrentService.cs`, `TorrentCatalog.cs`, `TorrentCatalogStore.cs`

**Проблема**
Persistence и runtime orchestration слишком близко расположены и читаются как единый поток.

**Риски**
- баги на перезапуске приложения
- сложность воспроизведения ошибок restore/load/save
- повышенная хрупкость shutdown сценариев

**Что делать**
Выделить явный слой хранения/восстановления состояния и отделить его от runtime-команд.

**Ожидаемый эффект**
Проще отлаживать lifecycle, меньше сюрпризов после рестарта приложения.

## High

### TD-005. `TorrentViewModel` содержит orchestration, а не только presentation logic
**Статус:** todo  
**Файлы:** `src/NiTorrent.Presentation/Features/Torrents/TorrentViewModel.cs`

**Проблема**
ViewModel совмещает:
- UI state
- работу с командами
- orchestration add/remove/start/pause
- подписки и синхронизацию коллекции
- агрегирующую логику

**Почему это плохо**
- ViewModel быстро разрастается
- UI-изменения цепляют сценарную логику
- тяжело тестировать без UI-контекста

**Что делать**
Постепенно выносить сценарии в Application:
- `AddTorrentUseCase`
- `PauseTorrentUseCase`
- `ResumeTorrentUseCase`
- `RemoveTorrentUseCase`
- `OpenTorrentFolderUseCase`

**Ожидаемый эффект**
Тоньше UI-слой и понятнее границы ответственности.

---

### TD-006. Слой Application пока слишком тонкий

**Файлы:** `src/NiTorrent.Application/*`

**Что делать**
Добавить прикладные сценарии и фасады для операций над торрентами и настройками.

**Ожидаемый эффект**
Появится центральное место для сценариев, проще писать тесты и понимать поток данных.

### TD-009. Риск смешивания platform-specific логики и более общих сценариев
**Статус:** todo  
**Файлы:** `NiTorrent.App`, `NiTorrent.Presentation`

**Проблема**

- App = platform adapters

**Ожидаемый эффект**

кнопка -> команда -> ViewModel -> сервис -> runtime -> persistence.

**Что делать**

- Add torrent
- Resume torrent
- Remove torrent
- Restore state on startup

**Ожидаемый эффект**
Ускорение онбординга и проще ревью крупных изменений.

## Medium

### TD-011. Недостаточно формализованы границы модулей `Settings`
**Статус:** todo

**Проблема**
Настройки распределены между Domain/Application/Infrastructure/App и это нормально, но пока не очень явно описано, где заканчиваются preferences, а где начинается platform storage.

**Что делать**
Сделать отдельный раздел в архитектуре:
- что считается domain-level setting
- что считается app preference
- кто отвечает за загрузку/сохранение

**Ожидаемый эффект**
Меньше путаницы при добавлении новых настроек.

### TD-012. Недостаточно изолированы pure transformation/mapping части
**Статус:** todo

**Проблема**
Часть преобразований данных живёт рядом с тяжелыми operational методами.

**Что делать**
Выделять отдельно:
- snapshot mapping
- DTO/domain conversion

**Ожидаемый эффект**
Проще тестировать и безопаснее рефакторить.

### TD-013. Уязвимость к росту количества условных веток в ViewModel и startup orchestration
**Статус:** todo

**Проблема**

**Ожидаемый эффект**
Более предсказуемый рост кода.

### TD-014. Нет явного списка performance hot spots и способов их проверки
**Статус:** todo

**Проблема**
Цель "улучшить производительность" есть, но нет списка, где и чем её мерить.

**Что делать**
Для начала зафиксировать 3–5 замеров:
- холодный старт приложения
- время восстановления списка торрентов

- стоимость частых snapshot/update операций
- поведение при нескольких активных торрент-сессиях

**Ожидаемый эффект**

### TD-015. Возможное дублирование сценарных знаний между App, Presentation и Infrastructure
**Статус:** todo

**Проблема**
Если сценарии не централизовать в Application, знание о том, как должен идти поток, начинает дублироваться по слоям.

**Что делать**
Проверять каждый новый сценарий вопросом:
- где его истинный owner?
- не дублируется ли он уже в другом слое?

**Ожидаемый эффект**
Меньше расхождений в поведении и ниже риск скрытых багов.

## Что не надо делать сразу
Чтобы не закопаться, пока **не стоит** начинать с такого:
- глобальное переименование файлов и пространств имён
- массовая перестройка всех директорий
- переписывание всего слоя Presentation целиком
- полная замена MonoTorrent integration за один этап
- косметическая чистка без снижения риска

---

## Definition of done для задач техдолга
Любая крупная задача техдолга считается завершённой только если:
- ответственность реально вынесена в правильный слой
- старый путь удалён или явно помечен как временный
- обновлена документация в `docs/project-map/`
- пройден `REGRESSION_CHECKLIST.md`
- новый код понятнее старого не только автору, но и следующему разработчику
