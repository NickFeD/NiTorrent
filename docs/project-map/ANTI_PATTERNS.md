# NiTorrent — ANTI_PATTERNS

## Зачем нужен этот файл
Этот документ фиксирует архитектурные и организационные ошибки, которые **нельзя размножать дальше**.

Он нужен для двух задач:
- чтобы в ходе рефакторинга не тащить старые проблемы в новые компоненты
- чтобы новые PR не усиливали связанность и сложность проекта

Документ не про «идеальную архитектуру», а про практическое правило:
**если изменение создаёт один из анти-паттернов ниже, его нужно переделать до merge**.

---

## Главные анти-паттерны проекта

## 1. God class / слишком умный класс
### Как выглядит
Один класс знает и делает слишком много:
- хранит состояние
- оркестрирует сценарии
- ходит во внешние зависимости
- управляет жизненным циклом
- делает mapping
- логирует ошибки
- принимает решения за UI или application layer

### Где это уже видно
- `src/NiTorrent.Infrastructure/Torrents/MonoTorrentService.cs`
- частично `src/NiTorrent.App/App.xaml.cs`
- частично `src/NiTorrent.Presentation/Features/Torrents/TorrentViewModel.cs`

### Почему это плохо
- трудно тестировать
- трудно читать новичку
- правка в одном месте ломает несколько сценариев сразу
- сложнее изолировать баги

### Что делать вместо этого
Разделять ответственность по ролям:
- lifecycle
- commands
- recovery
- snapshot building
- UI state
- orchestration

---

## 2. Composition root превращается в business coordinator
### Как выглядит
`App.xaml.cs` или другой entry point начинает:
- принимать архитектурные решения
- оркестрировать use case
- содержать ветвистую сценарную логику
- напрямую управлять инфраструктурой

### Почему это плохо
- startup становится хрупким
- сложно отлаживать activation / shutdown
- точка входа перестаёт быть понятной

### Что делать вместо этого
В `App.xaml.cs` оставлять только:
- bootstrap
- DI
- подписки на верхнеуровневые события
- делегирование в отдельные сервисы

Логику переносить в:
- `AppActivationService`
- `WindowCoordinator`
- `ShutdownCoordinator`
- application/use-case слой

---

## 3. Толстая ViewModel
### Как выглядит
ViewModel начинает делать больше, чем нужно UI:
- общается с несколькими сервисами сразу
- держит orchestration нескольких сценариев
- принимает инфраструктурные решения
- знает про детали engine / storage / recovery
- содержит тяжёлую коллекционную и таймерную логику

### Где риск высокий
- `src/NiTorrent.Presentation/Features/Torrents/TorrentViewModel.cs`

### Почему это плохо
- растёт риск UI-регрессий
- сложно покрывать тестами
- становится неясно, где presentation, а где application logic

### Что делать вместо этого
Во ViewModel оставлять:
- observable state
- команды уровня UI
- вызов application/use-case сервисов
- преобразование модели в вид для экрана

Не держать во ViewModel:
- recovery logic
- lifecycle logic
- работу с файловой системой
- знания о MonoTorrent internals

---

## 4. Infrastructure leakage в Presentation
### Как выглядит
Presentation начинает зависеть от:
- MonoTorrent-типов
- platform-specific API
- файловой системы
- конкретных storage реализаций

### Почему это плохо
- Presentation становится трудно тестировать
- слой теряет переносимость и предсказуемость
- границы архитектуры стираются

### Что делать вместо этого
Presentation должна работать только с:
- domain/application моделями
- интерфейсами из `NiTorrent.Application`
- своими ViewModel и mapping на UI

---

## 5. Platform-specific код просачивается выше App
### Как выглядит
WinUI/Windows-специфика появляется в:
- `NiTorrent.Presentation`
- `NiTorrent.Application`
- `NiTorrent.Domain`

Примеры:
- `DispatcherQueue`, `Window`, `AppWindow`, `FileOpenPicker`, `LaunchActivatedEventArgs`
- прямые обращения к WinRT API вне `NiTorrent.App`

### Почему это плохо
- архитектура перестаёт быть слоистой
- онбординг усложняется
- код сложнее тестировать и переиспользовать

### Что делать вместо этого
Оставлять platform-specific детали в `NiTorrent.App` и прятать их за адаптерами:
- `IDialogService`
- `IPickerHelper`
- `IUriLauncher`
- `IUpdateService`

---

## 6. Смешение pull и push модели без явных правил
### Как выглядит
Одна и та же зона состояния обновляется сразу несколькими способами:
- polling
- события
- ручной refresh
- периодические sync-операции

### Почему это плохо
- появляются гонки
- состояние в UI может обновляться неочевидно
- сложно понять источник истины

### Что делать вместо этого
Для каждой зоны выбрать и зафиксировать:
- кто источник истины
- кто публикует изменения
- кто подписывается
- какова частота обновления

Для торрентов особенно важно явно решить:
- snapshots приходят событием
- либо они запрашиваются по расписанию
- либо одно оборачивается в другое, но это должно быть описано явно

---

## 7. Неявные side effects
### Как выглядит
Метод по названию делает одно, а по факту ещё:
- сохраняет состояние
- запускает/останавливает менеджер
- публикует событие
- трогает UI
- пишет настройки

### Почему это плохо
- трудно предсказать последствия вызова
- сложно писать безопасные изменения
- повышается шанс регрессий

### Что делать вместо этого
- давать честные имена
- side effect'ы выносить в отдельные этапы
- делать сценарии явными
- документировать порядок действий

Пример хорошего направления:
- `AddTorrentAsync`
- `PersistCatalogAsync`
- `PublishSnapshotUpdate`

а не один метод, который делает всё сразу без явной границы

---

## 8. Скрытые temporal coupling зависимости
### Как выглядит
Код работает только если методы вызваны в «правильном» порядке, но это нигде не зафиксировано.

Примеры:
- нужно сначала инициализировать engine, потом создать window, потом поднять tray
- restore должен завершиться до подписки UI
- dispatcher должен быть доступен до публикации изменений

### Почему это плохо
- приложение становится хрупким при простом переносе кода
- баги проявляются только на старте или shutdown
- сложно объяснить порядок новичку

### Что делать вместо этого
- делать порядок явным в coordinator/service
- кодировать состояние через lifecycle-фазы
- защищать опасные пути guard-check'ами
- логировать startup/shutdown шаги

---

## 9. Слишком ранняя общая абстракция
### Как выглядит
Ради «красоты» создаются абстракции, которые пока не нужны:
- общий helper на всё подряд
- generic manager без чёткой ответственности
- базовые классы для пары use case
- util-папка для несвязанных вещей

### Почему это плохо
- проект начинает скрывать смысл за псевдо-универсальностью
- растёт сложность навигации
- новичок не понимает, где реальная ответственность

### Что делать вместо этого
Сначала выделять конкретные сценарные или инфраструктурные компоненты с ясной ролью.
Обобщать только после 2–3 реальных повторений.

---

## 10. Utility dump / папка «на всё»
### Как выглядит
Новые куски кода складываются в:
- `Helpers`
- `Utils`
- `Common`
- `Extensions`

без понятной доменной или слоевой принадлежности.

### Почему это плохо
- структура проекта размывается
- поиск ответственности становится дорогим
- со временем там накапливается хаос

### Что делать вместо этого
Каждый новый файл класть по ответственности:
- сценарий — в Application
- адаптер платформы — в App
- интеграция с движком — в Infrastructure
- UI state — в Presentation
- инварианты и модели — в Domain

---

## 11. Изменение поведения вперемешку с перемещением кода
### Как выглядит
В одном PR одновременно:
- переименование
- перенос файлов
- изменение алгоритма
- чистка форматирования
- новая логика

### Почему это плохо
- невозможно качественно ревьюить
- трудно искать источник багов
- почти невозможно безопасно откатить

### Что делать вместо этого
Разделять PR по типам:
1. подготовка
2. перенос без изменения поведения
3. подключение новой реализации
4. удаление старого пути

---

## 12. Долгоживущие временные адаптеры
### Как выглядит
На время миграции добавляется прослойка, но потом она остаётся навсегда.

Примеры:
- временный facade
- bridge для старой и новой модели
- костыльный mapper
- fallback код на случай старого сценария

### Почему это плохо
- система становится сложнее, чем была
- никто уже не понимает, что считается основным путём
- миграция никогда не заканчивается

### Что делать вместо этого
Для каждого временного адаптера сразу фиксировать:
- зачем он нужен
- какой код он заменяет
- при каком условии удаляется
- в каком PR/этапе должен быть удалён

---

## 13. Отсутствие явного владельца состояния
### Как выглядит
Непонятно, кто отвечает за правду по состоянию торрента:
- engine
- каталог
- snapshot
- UI-коллекция
- сохранённое состояние

### Почему это плохо
- легко получить рассинхрон
- восстановление после перезапуска становится ненадёжным
- баги плавают между слоями

### Что делать вместо этого
Для каждого типа состояния фиксировать:
- authoritative source
- read model
- persistence format
- момент обновления
- правила восстановления

---

## 14. Неразделённые ошибки: user-facing и technical
### Как выглядит
Одинаково обрабатываются:
- технические ошибки движка
- ошибки файловой системы
- отмена пользователем диалога
- бизнес-ошибки сценария

### Почему это плохо
- неудобный UX
- плохая диагностика
- логирование бесполезно для анализа релизной нестабильности

### Что делать вместо этого
Разделять хотя бы 3 класса ошибок:
- user-cancel / expected flow
- recoverable application errors
- critical technical failures

И по-разному их:
- логировать
- показывать в UI
- эскалировать

---

## 15. Рефакторинг без фиксации текущего поведения
### Как выглядит
Опасная зона переписывается «по ощущению», без smoke-проверки и without baseline.

### Почему это плохо
- становится непонятно, это улучшение или регрессия
- релизы дестабилизируются
- команда боится трогать код дальше

### Что делать вместо этого
Перед рефакторингом любой hot spot зоны:
- зафиксировать сценарии
- добавить ручной regression checklist
- включить базовое логирование
- только потом переносить ответственность

---

## Архитектурные запреты для новых PR
Ниже то, что нужно считать стоп-сигналом на ревью.

### Нельзя
- тащить WinUI API в `Presentation`, `Application` или `Domain`
- добавлять новые обязанности в `App.xaml.cs`
- расширять `MonoTorrentService`, если поведение можно вынести в отдельный компонент
- добавлять orchestration в ViewModel
- складывать новый код в безличные `Helpers`/`Utils`
- смешивать перенос, поведение и форматирование в одном PR
- оставлять временные адаптеры без плана удаления

### Можно
- выделять coordinator/use case/application service
- вводить узкие интерфейсы вместо расширения god class
- делать thin ViewModel и thin composition root
- описывать lifecycle и ownership состояния явно

---

## Что проверять на ревью
Перед merge полезно пройти короткий фильтр.

### Вопросы к PR
1. Этот PR уменьшает связанность или увеличивает её?
2. Не появился ли новый god class?
3. Не протекли ли platform-specific детали вверх по слоям?
4. Не стала ли ViewModel умнее, чем должна быть?
5. Явно ли видно, кто владеет состоянием и кто публикует обновления?
6. Можно ли объяснить новую структуру новичку за 2–3 минуты?
7. Есть ли план удаления временных прослоек?
8. Разделены ли структурные изменения и изменения поведения?

Если на 2 и более вопроса ответ тревожный — PR лучше переработать до merge.

## Самые важные анти-паттерны, которые надо давить первыми
Если сил мало, фокусироваться именно на этих четырёх:
1. god class в `MonoTorrentService`
2. перегруженный `App.xaml.cs`
3. толстая `TorrentViewModel`
4. протекание platform/infrastructure деталей через слои

Именно они сейчас сильнее всего бьют по:
- багам
- онбордингу
- стабильности релизов
- стоимости дальнейших изменений
