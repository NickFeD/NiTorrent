# NiTorrent — REGRESSION_CHECKLIST

Этот чеклист нужен для безопасного рефакторинга, проверки PR и подготовки релизов.

Главная цель: **не сломать рабочие пользовательские сценарии**, пока проект постепенно упрощается.

---

## Как пользоваться

Чеклист делится на 3 режима:
- **перед рефакторингом** — что проверить до начала изменений
- **после изменения модуля** — что проверить локально и на PR
- **перед релизом** — что проверить на сборке целиком

Не обязательно проходить весь список на каждый маленький PR.

Практичное правило:
- если менялся только UI — пройти UI + smoke
- если менялся startup / tray / activation — пройти startup + shell сценарии
- если менялся торрент-движок — пройти engine + restore + state transitions
- если менялись DI / composition root — пройти startup, add torrent, stop, exit, relaunch

---

## 1. Перед началом рефакторинга

### 1.1. Зафиксировать границы изменения
Перед работой ответить письменно:
- какой модуль меняем
- что именно улучшаем
- какое текущее поведение обязано сохраниться
- какие сценарии считаются критичными
- что в этот PR **не входит**

### 1.2. Убедиться, что есть точка отката
- проект собирается до изменений
- приложение запускается локально
- есть рабочая ветка / commit до начала изменений
- понятно, какие файлы были затронуты

### 1.3. Отметить риск-уровень изменения
**Низкий риск**
- перемещение кода без изменения поведения
- мелкий UI cleanup
- извлечение private-методов

**Средний риск**
- изменение ViewModel
- изменение DI-регистраций
- выделение новых сервисов / coordinator / facade

**Высокий риск**
- изменение `App.xaml.cs`
- изменение `MonoTorrentService`
- изменение жизненного цикла торрентов
- изменение восстановления состояния после перезапуска
- изменение shutdown / dispose / exit flow

Для high-risk PR желательно отдельно пройти полный smoke ниже.

---

## 2. Базовый smoke-check после любого заметного изменения

После каждого не совсем тривиального PR проверить минимум это:

- приложение собирается без ручных правок
- приложение запускается без исключений на старте
- главное окно открывается
- список торрентов отображается корректно
- приложение закрывается без зависания
- повторный запуск после закрытия работает

Если меняли startup, DI, инфраструктуру или torrent flow — этого недостаточно, нужен расширенный чеклист ниже.

---

## 3. Startup / Shell / App lifecycle

Проверять при изменениях в:
- `NiTorrent.App`
- `App.xaml.cs`
- single-instance логике
- tray поведении
- activation handling
- composition root / DI

### Проверки
- приложение запускается с чистого состояния
- приложение запускается повторно без конфликтов single-instance
- вторичный запуск не создаёт сломанную вторую копию
- главное окно показывается корректно
- закрытие окна работает ожидаемо
- сворачивание / hiding в tray работает как ожидалось
- восстановление окна из tray работает
- выход из tray полностью завершает приложение
- после полного выхода процесс не остаётся висеть в системе
- повторный запуск после полного выхода стабилен
- переданный `.torrent`-файл обрабатывается корректно
- запуск приложения через открытие `.torrent` не ломает activation flow

### На что смотреть отдельно
- нет ли зависания на shutdown
- нет ли двойной инициализации сервисов
- нет ли двойных подписок на события
- нет ли ошибок при dispose и освобождении ресурсов

---

## 4. Torrent engine / lifecycle / persistence

Проверять при изменениях в:
- `MonoTorrentService`
- storage / state persistence
- snapshot / restore
- add / start / pause / stop / remove сценариях

### 4.1. Добавление торрента
- торрент добавляется без исключения
- новый элемент появляется в списке
- имя / размер / базовые поля отображаются корректно
- duplicate scenario обработан предсказуемо
- не создаются битые записи состояния

### 4.2. Запуск и остановка
- старт торрента работает
- пауза работает
- stop работает
- повторный start после pause/stop работает
- состояние в UI соответствует реальному состоянию движка
- быстрые повторные нажатия не ломают lifecycle

### 4.3. Удаление
- remove удаляет торрент из списка
- remove не оставляет мусорные state-записи
- удаление не ломает остальные активные торренты
- после удаления и перезапуска приложение не пытается восстановить удалённый торрент

### 4.4. Восстановление состояния после перезапуска
- после закрытия приложения список корректно восстанавливается
- восстановленные торренты имеют ожидаемый статус
- не появляются дубли
- не теряются базовые метаданные
- приложение не падает на битом или частично заполненном state

### 4.5. Параллельность и устойчивость
- несколько быстрых команд подряд не приводят к гонкам
- нет deadlock / зависания при add-start-pause-remove последовательностях
- нет ошибок из-за `SemaphoreSlim` / async-переходов
- исключение в одном торренте не валит всё приложение

## 5. UI / Presentation / ViewModel

Проверять при изменениях в:
- `TorrentViewModel`
- presentation models
- commands
- subscriptions / event handlers
- коллекциях и биндингах

### Проверки
- список торрентов обновляется без визуальных артефактов
- команды add/start/pause/remove доступны в правильных состояниях
- кнопки не остаются навсегда disabled/enabled по ошибке
- повторное открытие окна не создаёт дубли подписок
- данные в UI не рассинхронизируются с backend state
- массовое обновление списка не вызывает подвисаний UI
- удалённый элемент исчезает из UI один раз, без фантомных дублей

### На что смотреть отдельно
- нет ли тяжёлой логики прямо в ViewModel
- нет ли вызовов инфраструктуры напрямую из UI-слоя
- нет ли утечек через event subscriptions

---

## 6. Производительность

Проверять после изменений, которые касаются:
- больших коллекций
- частых обновлений статуса
- polling / events
- сериализации состояния
- startup/shutdown flow

### Проверки
- время запуска не стало заметно хуже
- UI не подвисает при обновлении списка
- нет избыточных полных перерисовок коллекции
- нет лишних пересозданий snapshot / DTO / VM
- нет частого лишнего сохранения состояния на каждый мелкий шаг
- CPU не растёт без причины в idle
- память не продолжает расти после add/remove/start/stop циклов

### Минимальные сценарии
- открыть приложение с пустым состоянием
- открыть приложение с уже восстановленным списком
- добавить торрент и понаблюдать UI
- несколько раз подряд start/pause/stop
- закрыть и снова открыть приложение

---

## 7. Наблюдаемость и обработка ошибок

Проверять, если менялась обработка исключений, startup, engine, persistence.

### Проверки
- ошибка логируется достаточно понятно
- в логах можно понять, на каком этапе упало приложение
- ошибки пользователя и системные ошибки не смешиваются в один непонятный текст
- исключение не проглатывается молча
- авария в инфраструктуре не приводит к тихо сломанному UI

### Минимум, который должен быть понятен из логов
- старт приложения
- инициализация хоста
- инициализация torrent engine
- add/start/pause/stop/remove
- restore state
- shutdown / dispose

---

## 8. PR checklist

Этот раздел можно использовать как шаблон в описании PR.

### Что проверить автору
- изменение ограничено одной целью
- поведение не менялось без явного описания
- временные адаптеры и обходные решения отмечены
- новые классы названы по роли, а не абстрактно
- DI-регистрации не дублируются
- публичные интерфейсы не раздуты лишними методами
- нет случайного переноса бизнес-логики в UI
- нет прямых зависимостей Presentation -> Infrastructure, если их можно избежать
- старый код удалён там, где миграция завершена
- обновлена документация в `docs/project-map/`, если менялись границы модулей

### Что проверить ревьюеру
- действительно ли PR решает одну задачу
- можно ли читать изменения по шагам
- есть ли смешение рефакторинга и изменения поведения
- есть ли скрытые lifecycle-изменения
- не появилась ли новая связность между слоями
- не стал ли `App` / `ViewModel` / `MonoTorrentService` ещё толще

## 9. Release checklist

Проходить перед выпуском сборки, если были изменения в startup, engine, storage или UI flows.

### Сборка и запуск
- clean build проходит
- release build проходит
- приложение запускается на чистой машине / чистом профиле
- нет missing dependency / config ошибок

### Критичные сценарии пользователя
- первый запуск приложения работает
- добавление `.torrent` работает
- список отображается корректно
- старт / пауза / stop / remove работают
- закрытие и повторный запуск работают
- восстановление состояния работает
- tray / exit flow работают

### Устойчивость
- приложение не оставляет зависший процесс после закрытия
- не появляется повреждённое состояние после аварийного сценария
- ошибки не делают приложение неуправляемым

### Документация и сопровождение
- обновлены архитектурные заметки при изменении границ
- обновлён план рефакторинга, если закрыт этап
- описаны known limitations, если временное решение оставлено в релизе

## 10. Что тестировать в первую очередь именно в этом проекте

Для текущей структуры проекта самый полезный порядок проверки такой:

### При изменении `App.xaml.cs`
1. запуск приложения
2. повторный запуск / single-instance
3. tray
4. exit
5. relaunch
6. открытие `.torrent`

### При изменении `MonoTorrentService`
1. add torrent
2. start
3. pause
4. stop
5. remove
6. close app
7. relaunch
8. restore state

### При изменении `TorrentViewModel`
1. список загружается
2. команды доступны в правильном состоянии
3. UI обновляется после действий
4. удаление отражается корректно
5. повторное открытие не создаёт дубли

## 11. Антипаттерны, после которых нужен повторный прогон smoke

Если в PR есть что-то из списка ниже — прогонять расширенный smoke обязательно:
- перенос логики между слоями
- изменение async flow
- изменение DI lifetime
- изменение event subscriptions
- изменение shutdown/dispose
- изменение state restore
- изменение структуры коллекций в UI
- удаление старых guard/check блоков без явной замены

---

## 12. Минимальный ручной сценарий на 5 минут

Если совсем мало времени, пройти хотя бы это:

1. Запустить приложение.
2. Убедиться, что главное окно открылось без ошибок.
3. Добавить торрент.
4. Проверить, что элемент появился в списке.
5. Запустить торрент.
6. Поставить на паузу.
7. Снова запустить.
8. Закрыть приложение.
9. Запустить приложение повторно.
10. Убедиться, что состояние восстановилось корректно.
11. Удалить торрент.
12. Полностью выйти из приложения.
13. Проверить, что процесс не висит.

---

## 13. Рекомендуемое развитие после этого чеклиста

Когда начнётся активный рефакторинг, этот документ стоит усилить:
- превратить часть smoke-проверок в automated integration tests
- добавить PR template на основе раздела 8
- завести маленький regression suite на startup + torrent lifecycle + restore
- прикрепить чеклист к релизному процессу

Тогда рефакторинг перестанет быть “опасной ручной операцией” и станет управляемой серией изменений.
