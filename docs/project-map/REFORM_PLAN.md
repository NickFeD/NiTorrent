# NiTorrent — REFORM_PLAN

## Зачем этот план
План собран под твои цели:
- снизить количество багов
- упростить онбординг
- убрать просадки производительности
- сделать релизы стабильнее

Главная идея: **не переписывать всё**, а идти короткими этапами, где каждый шаг уменьшает риск и оставляет проект в рабочем состоянии.

---

## Что сейчас самое рискованное

### 1. `src/NiTorrent.Infrastructure/Torrents/MonoTorrentService.cs`
Самая крупная техническая точка риска.

Сейчас в одном классе смешаны:
- lifecycle торрент-движка
- восстановление состояния
- add/start/pause/stop/remove
- сохранение состояния
- построение snapshot
- синхронизация через `SemaphoreSlim`
- часть orchestration-логики

Последствия:
- высокий шанс багов при правках
- трудно локализовать причины нестабильных релизов
- сложно тестировать
- тяжело объяснять новичку, где кончается инфраструктура и начинается сценарная логика

### 2. `src/NiTorrent.App/App.xaml.cs`
Слишком много обязанностей в точке входа.

Сейчас здесь совмещены:
- bootstrap host/DI
- single-instance поведение
- tray lifecycle
- window lifecycle
- activation handling для `.torrent`
- запуск и инициализация торрент-сервиса

Последствия:
- неудобный startup flow
- риск регрессий при изменении запуска/выхода
- тяжело читать и быстро понять границы приложения

### 3. `src/NiTorrent.Presentation/Features/Torrents/TorrentViewModel.cs`
Важная UI-VM, но в ней уже есть orchestration, которое лучше жить не должно.

Сейчас там смешаны:
- подписки на сервис
- обновление UI-коллекции
- команды start/pause/remove/add
- агрегация состояний
- часть системных действий

Последствия:
- ViewModel становится точкой накопления логики
- сложно покрывать тестами
- любое изменение UI рискует задеть бизнес-сценарий

## Общие правила рефакторинга
Перед стартом договориться о правилах и придерживаться их на каждом PR.

### Правило 1. Один PR — одна цель
Не смешивать в одном PR:
- перенос файлов
- изменение поведения
- переименование
- оптимизацию
- архитектурную декомпозицию

### Правило 2. Сначала страховка, потом перенос
Перед изменением опасной зоны:
- зафиксировать текущее поведение
- добавить smoke/integration checks на критичные сценарии
- только после этого переносить ответственность между классами

### Правило 3. Сначала уменьшать связанность, потом чистить стиль
Приоритет всегда у изменений, которые делают систему проще для безопасных изменений.

### Правило 4. Не чинить всё, что увидели
Если в задаче открылись ещё 5 проблем:
- чиним только то, что мешает текущей цели
- остальные фиксируем в backlog

### Правило 5. Старое удаляется только после переключения
Подход:
- выделили интерфейс или новый компонент
- подключили рядом со старым
- переключили use case
- проверили
- удалили старый путь

## Этап 2. Разгрузить `App.xaml.cs`
Цель: стабилизировать startup/shutdown и упростить понимание входной точки.

### Что вынести
Из `App.xaml.cs` лучше выделить отдельные компоненты:
- `AppBootstrapper` — запуск host, инициализация сервисов, startup orchestration
- `AppActivationService` — обработка activation и `.torrent` файлов
- `MainWindowLifecycle` или `WindowCoordinator` — создание, показ, hide-to-tray, closing behavior
- `ShutdownCoordinator` — сохранение и остановка приложения

### Порядок действий
1. Вынести только чтение/обработку activation в отдельный сервис.
2. Вынести tray/window lifecycle.
3. Вынести startup orchestration (`InitializeApp`, `InitializeTorrentEngineAsync`).
4. Оставить в `App.xaml.cs` только composition root и делегирование.

### Что должно остаться в `App.xaml.cs`
- создание host
- вызов `InitializeComponent()`
- связывание верхнеуровневых событий
- делегирование в специализированные сервисы

### Ожидаемый эффект
- проще сопровождать startup
- меньше регрессий в релизах
- легче понять приложение новичку

## Этап 3. Разрезать `MonoTorrentService` на роли
Цель: это главный этап по багам, стабильности и частично производительности.

### Целевое разбиение
Рекомендуемое направление декомпозиции:

- `TorrentEngineLifecycle`  
  отвечает за создание/инициализацию `ClientEngine`, загрузку/остановку состояния

- `TorrentSessionRepository` или `TorrentStateRepository`  
  отвечает за чтение/запись engine state и каталога

- `TorrentCommandExecutor`  
  отвечает за `Add/Start/Pause/Stop/Remove`

- `TorrentSnapshotFactory`  
  отвечает за построение `TorrentSnapshot`

- `TorrentRuntimeRegistry`  
  отвечает за `TorrentId <-> TorrentManager`

- `TorrentStartupRecovery`  
  отвечает за восстановление менеджеров и синхронизацию с каталогом

Сам `MonoTorrentService` после этого должен стать тонким фасадом над этими компонентами.

### Как резать безопасно
1. Сначала вынести pure/helper части без изменения поведения.
2. Потом вынести snapshot building.
3. Потом вынести recovery/load/save.
4. Потом вынести command execution.
5. В конце оставить `MonoTorrentService` только как реализацию интерфейса `ITorrentService`.

### На что обратить внимание
- не ломать семантику `SemaphoreSlim`
- не менять одновременно и структуру, и поведение
- сохранять совместимость с `ITorrentService`
- на каждом шаге проверять startup/load/save/pause/remove

### Ожидаемый эффект
- ниже шанс багов при изменениях
- проще искать причины ошибок
- проще читать код
- проще писать тесты на отдельные куски

## Этап 4. Перенести orchestration из `TorrentViewModel`
Цель: облегчить Presentation и сделать сценарии читаемыми.

### Что должно уйти из VM
Из `TorrentViewModel` лучше постепенно вынести:
- orchestration add/start/pause/remove
- обработку сложных последовательностей действий
- синхронизацию нескольких сервисов
- часть агрегирующей логики, не относящейся к UI state

### Куда переносить
В `NiTorrent.Application` стоит добавить сценарные классы, например:
- `AddTorrentUseCase`
- `PauseTorrentUseCase`
- `ResumeTorrentUseCase`
- `RemoveTorrentUseCase`
- `GetTorrentListUseCase`
- `OpenTorrentFolderUseCase` или аналогичный command handler

### Роль `TorrentViewModel` после рефакторинга
- хранит observable state
- вызывает application use cases
- маппит результат в UI-модель
- не знает деталей MonoTorrent и platform logic

### Ожидаемый эффект
- проще онбординг
- меньше логики на уровне UI
- выше тестопригодность

## Этап 5. Нормализовать события и обновление состояния
Цель: уменьшить нестабильность и странные баги синхронизации.

### Что видно сейчас
В `ITorrentService` есть:
- `Loaded`
- `UptateTorrent`
- `GetAll`
- `TryGet`
- `UpdateTorrent()`

В этом API смешаны push и pull подходы, а имя `UptateTorrent` содержит опечатку.

### Что рекомендуется
Постепенно привести API к более понятной модели:
- одно событие состояния списка или потока изменений
- единый контракт обновления snapshots
- явная семантика загрузки initial state

### Минимальный безопасный шаг
1. Не ломая совместимость, обернуть текущие события в внутренний coordinator.
2. Потом ввести новое имя и новый контракт.
3. Старый API удалить только после перевода Presentation.

### Ожидаемый эффект
- меньше хрупких подписок
- меньше случайной дублирующей логики в ViewModel
- легче отлаживать обновление UI

---

## Этап 6. Навести порядок в сохранении состояния
Цель: снизить баги на рестарте и нестабильность релизов.

### Что проверить
- где именно сохраняется engine state
- где именно сохраняется catalog
- нет ли гонок между `SaveAsync`, закрытием окна и выходом
- корректно ли согласованы `ShouldRun`, `Phase`, `AddedAtUtc`
- что происходит при исключениях в shutdown

### Рекомендуемые шаги
1. Вынести policy сохранения в отдельный компонент.
2. Разделить `force save` и `debounced save`.
3. Явно описать порядок shutdown:
   1. перестать принимать новые команды
   2. сохранить каталог
   3. сохранить engine state
   4. остановить hosted services
   5. завершить host

### Ожидаемый эффект
- меньше повреждённых состояний
- стабильнее запуск после предыдущего выхода
- понятнее сценарий релиза и ручной проверки

## Этап 7. Отдельно пройтись по производительности
Цель: не “оптимизировать всё подряд”, а убрать реальные узкие места.

### Что измерить сначала
- время startup до готовности UI
- время startup до готовности торрент-движка
- частоту и стоимость `UpdateTorrent()`
- размер и частоту snapshot updates
- сколько раз полностью пересобирается UI-коллекция
- есть ли лишние дисковые операции при сохранении

### Вероятные зоны улучшения
- уменьшить объём работы на каждом тике `TorrentMonitor`
- не пересчитывать всё состояние, если изменились только отдельные торренты
- минимизировать лишние сериализации каталога
- не выполнять тяжёлую работу в UI thread

### Важно
Оптимизацию делать **после** разрезания ответственности. Иначе проект станет быстрее, но ещё менее поддерживаемым.
